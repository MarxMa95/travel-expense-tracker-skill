# 业务规则清单

## 核心规则

| 规则 | 说明 |
|------|------|
| 数据存储 | 直接写入飞书多维表格，不经过本地数据库 |
| 多维表创建 | 必须由 Bot 创建，确保群成员可正常写入 |
| API 调用身份 | 使用 Tenant Access Token，Bot 身份调用 |
| 群成员权限 | Bot 创建后需为所有群成员开启 `full_access` 权限 |
| 付款人规则 | 发布消息的用户 = 默认付款人 |
| 交易状态过滤 | 交易关闭、已取消、已退款、未支付的订单不计入 |
| 默认消费类型 | 公共 AA |
| 默认币种 | CNY |
| 时间处理 | 无明确时间时用消息发送时间 |
| 住宿处理 | 按入住天数拆分，每晚一条，00:00 入账 |
| 信息不全 | @发图者请求补充，不直接落表 |
| 日报规则 | 只发送状态为“进行中”的项目日报 |
| 项目状态 | 每日 0:00 自动更新（未开始/进行中/已结束） |
| 多项目冲突 | 优先用当前群绑定项目，否则追问用户 |
| 修改记录 | 默认定位最近一笔；有歧义时必须列候选 |

## 交易状态过滤

以下状态的订单不计入统计，也不写入多维表：

- 交易关闭
- 已取消
- 已退款
- 未支付

## 住宿拆分规则

1. 必须提取入住日期和离店日期
2. 入住天数 = `离店日期 - 入住日期`
3. 每晚金额 = 总金额 / 天数
4. 每晚生成一条记录，时间设为 `00:00`
5. 如果信息不全，`@` 用户确认，不直接落表
6. 如果总金额不能整除，可保留两位小数；最后一晚用于吸收 rounding 差额更稳妥

## 项目匹配规则

按以下顺序选择项目：

1. 当前群已绑定且状态为“进行中”的项目
2. 当前日期命中的唯一“进行中”项目
3. 若存在多个候选，向用户确认
4. 若没有候选，引导创建新项目

## 日报发送规则

1. 只发送状态为“进行中”的项目
2. 每日 22:00 触发
3. 多项目时分别生成日报
4. 包含：今日消费、类别分布、明细、累计统计、付款人分布
5. 若当日无消费，也发送“今日无消费记录”的日报

## 修改记录规则

支持以下变更：

- 金额
- 消费时间
- 类别
- 消费类型
- 商户

不应在未确认的情况下批量修改多条记录。

## 权限配置规范

### Bot 创建多维表流程

1. Bot 调用 API 创建多维表
2. Bot 创建项目管理表和消费明细表
3. Bot 添加所有字段
4. Bot 为群成员开启 `full_access` 权限

### 添加成员权限

```http
POST /open-apis/drive/v1/permissions/{app_token}/members?type=bitable
```

```json
{
  "member_type": "openid",
  "member_id": "ou_xxx",
  "perm": "full_access"
}
```
